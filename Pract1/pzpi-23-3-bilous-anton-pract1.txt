                        3 ОПИС ВИКОНАНОЇ РОБОТИ

     3.1 Загальні відомості про шаблони проєктування

     Шаблон (патерн) проєктування – це типове рішення для певної проблеми,
що часто виникає під час проєктування програмного забезпечення. Патерн не є
конкретним фрагментом коду, а описує загальну концепцію розв’язання задачі, яку
можна пристосувати до конкретного контексту.
     У 1994 році Еріх Гамма, Річард Гелм, Ральф Джонсон та Джон Вліссідес
(«банда чотирьох», GoF) систематизували 23 патерни та поділили їх на три категорії:
      –   Породжувальні (Creational) – описують механізми створення об’єктів:
          Singleton, Factory Method, Abstract Factory, Builder, Prototype.
      –   Структурні (Structural) – описують способи складання об’єктів у більші
          структури: Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy.
      –   Поведінкові (Behavioral) – описують взаємодію між об’єктами та
          розподіл відповідальностей: Chain of Responsibility, Command, Iterator,
          Mediator, Memento, Observer, State, Strategy, Template Method, Visitor.
     Патерн Strategy належить до категорії поведінкових патернів.

     3.2 Опис обраного патерна

     Назва патерна: Strategy (Стратегія).
     Тип шаблона: поведінковий.
     Основне призначення: визначає сімейство алгоритмів, інкапсулює кожен із
них та робить їх взаємозамінними. Патерн дозволяє змінювати алгоритм незалежно
від клієнтів, що ним користуються.
     Передумови застосування:
      –   наявні кілька варіантів поведінки або алгоритму, що різняться лише в
          деталях реалізації;
      –   необхідно замінювати алгоритм під час виконання програми;
      –   існує велика кількість умовних операторів («if»/«match»), що обирають
          одну з кількох схожих реалізацій;
                                                                                     5
           –   необхідно ізолювати логіку алгоритму від коду клієнта для спрощення
               тестування.
           Переваги:
           –   відповідає принципу відкритості/закритості (Open/Closed Principle): нова
               стратегія додається без зміни наявного коду;
           –   кожна стратегія ізольована у власному типі, що спрощує модульне
               тестування;
           –   усуває розгалужені умовні конструкції у коді клієнта;
           –   дозволяє підміняти реалізацію під час виконання (за динамічної
               диспетчеризації).
           Недоліки:
           –   клієнт повинен знати про існуючі конкретні стратегії та самостійно
               обирати потрібну;
           –   використання патерна може бути надлишковим, якщо алгоритмів лише
               два і вони рідко змінюються;
           –   за динамічної диспетчеризації виникають витрати на heap-алокацію та
               непряму адресацію через таблицю віртуальних методів (vtable).

           3.3 Структура патерна

           Патерн Strategy складається з трьох структурних елементів.
           Context (Контекст) – клас або тип, що зберігає посилання на об’єкт стратегії
та делегує їй виконання алгоритму. Контекст не залежить від конкретних реалізацій
стратегії – він взаємодіє лише через спільний інтерфейс.
           Strategy (Стратегія) – інтерфейс або абстрактний клас, що визначає спільний
контракт для всіх конкретних стратегій. В об’єктно-орієнтованих мовах це зазвичай
абстрактний клас або інтерфейс. У мові Rust роль цього елемента виконує типаж
(trait).
           ConcreteStrategy (Конкретна стратегія) – конкретна реалізація алгоритму, що
відповідає контракту інтерфейсу Strategy. Кожна конкретна стратегія інкапсулює
один варіант алгоритму.
           Взаємодія між учасниками відбувається таким чином: клієнт створює об’єкт
конкретної стратегії та передає його контексту. Контекст зберігає стратегію та у
                                                                                     6
потрібний момент делегує їй виконання через виклик методу інтерфейсу. При цьому
контекст не знає, яка саме конкретна стратегія виконується.

      3.4 Реалізація мовою Rust

      У мові Rust інтерфейс Strategy природно реалізується через типаж (trait). Rust
надає два механізми поліморфізму: статичну диспетчеризацію через узагальнені
типи (generics) та динамічну диспетчеризацію через типові об’єкти (trait objects).

      3.4.1 Оголошення типажу та конкретних стратегій

      Типаж «SortStrategy» визначає спільний контракт для всіх стратегій
сортування. Конкретні стратегії «BubbleSort» та «QuickSort» реалізують цей типаж.

 1   trait SortStrategy {
 2       fn sort(&self, data: &mut Vec<i32>);
 3   }
 4
 5   struct BubbleSort;
 6
 7   impl SortStrategy for BubbleSort {
 8       fn sort(&self, data: &mut Vec<i32>) {
 9           let n = data.len();
10           for i in 0..n {
11               for j in 0..n - 1 - i {
12                   if data[j] > data[j + 1] {
13                       data.swap(j, j + 1);
14                   }
15               }
16           }
17       }
18   }
19
20   struct QuickSort;
21
22   impl SortStrategy for QuickSort {
23       fn sort(&self, data: &mut Vec<i32>) {
24           data.sort_unstable();
25       }
26   }


      3.4.2 Статична диспетчеризація

      За статичної диспетчеризації тип стратегії є параметром узагальненого
типу «Sorter<S>». Компілятор виконує мономорфізацію: для кожного конкретного
                                                                              7
типу «S» генерується окрема спеціалізація «Sorter». Виклик методу відбувається
безпосередньо, без додаткових витрат у runtime.

 1   struct Sorter<S: SortStrategy> {
 2       strategy: S,
 3   }
 4
 5   impl<S: SortStrategy> Sorter<S> {
 6       fn new(strategy: S) -> Self {
 7           Self { strategy }
 8       }
 9
10         fn run(&self, data: &mut Vec<i32>) {
11             self.strategy.sort(data);
12         }
13   }
14
15   fn main() {
16       let mut data = vec![5, 2, 8, 1, 9, 3];
17       let sorter = Sorter::new(BubbleSort);
18       sorter.run(&mut data);
19       println!("{:?}", data); // [1, 2, 3, 5, 8, 9]
20   }


         3.4.3 Динамічна диспетчеризація

         За динамічної диспетчеризації стратегія зберігається як типовий об’єкт
«Box<dyn SortStrategy>». Тип конкретної стратегії визначається під час виконання
програми, а виклик методу відбувається через vtable. Такий підхід дозволяє
підміняти стратегію під час виконання та зберігати різнотипні стратегії в одній
колекції.

 1   struct DynSorter {
 2       strategy: Box<dyn SortStrategy>,
 3   }
 4
 5   impl DynSorter {
 6       fn new(strategy: Box<dyn SortStrategy>) -> Self {
 7           Self { strategy }
 8       }
 9
10         fn set_strategy(&mut self, strategy: Box<dyn SortStrategy>) {
11             self.strategy = strategy;
12         }
13
14         fn run(&self, data: &mut Vec<i32>) {
15             self.strategy.sort(data);
16         }
17   }
                                                                                     8

18
19   fn main() {
20       let mut data = vec![5, 2, 8, 1, 9, 3];
21       let mut sorter = DynSorter::new(Box::new(BubbleSort));
22       sorter.run(&mut data);

24           data = vec![7, 3, 6, 1, 4];
25           sorter.set_strategy(Box::new(QuickSort)); // підміна у runtime
26           sorter.run(&mut data);
27           println!("{:?}", data); // [1, 3, 4, 6, 7]
28   }


         3.4.4 Патерн Strategy у стандартній бібліотеці Rust

         Патерн Strategy присутній у стандартній бібліотеці Rust у вигляді типажів, що
параметризують поведінку:
         –    «std::cmp::Ord» та замикання у «Vec::sort_by» – стратегія порівняння
              елементів при сортуванні;
         –    «std::io::Write» – стратегія запису байтів; одна функція може писати у
              файл, стандартний вивід або буфер у пам’яті без зміни власного коду;
         –    «std::hash::BuildHasher» – стратегія гешування для колекцій «HashMap»
              та «HashSet»; за замовчуванням використовується SipHash, але її можна
              замінити на іншу реалізацію.

      –   досліджено патерн проєктування Strategy, що належить до категорії

      –   проаналізовано структуру патерна: виокремлено учасників Context,

      –   реалізовано приклади програмного коду мовою Rust із використанням

          узагальнені типи (generics) та динамічної диспетчеризації через типові
          об’єкти («Box<dyn Trait>»);
      –   показано, що статична диспетчеризація забезпечує нульові витрати у
          runtime (zero-cost abstractions), тоді як динамічна – гнучкість підміни
          стратегії під час виконання;
      –   продемонстровано присутність патерна у стандартній бібліотеці Rust
          через типажі «Write», «BuildHasher» та замикання у методах сортування;
      –   підготовлено презентацію (13 слайдів) та відеодоповідь.
     Застосування патерна Strategy підвищує гнучкість і підтримуваність
програмного забезпечення, сприяє дотриманню принципу відкритості/закритості та
спрощує ізольоване тестування алгоритмів.

Реалізація в Rust: динамічна диспетчеризація
dyn Trait — vtable, стратегія обирається у runtime

// Стратегія зберігається як trait object
struct Sorter {
    strategy: Box<dyn SortStrategy>,                                    Box<dyn T> — heap-алокований trait
}                                                                       object з vtable
impl Sorter {
    fn new(s: Box<dyn SortStrategy>) -> Self {
        Sorter { strategy: s }
    }

    fn set_strategy(&mut self, s: Box<dyn SortStrategy>) {              Стратегію можна підмінити у runtime
        self.strategy = s;       // підміна під час виконання           (set_strategy)
    }

    fn run(&self, data: &mut Vec<i32>) {
        self.strategy.sort(data); // виклик через vtable
    }
}

  Strategy у стандартній бібліотеці Rust
  Патерн вбудований у мову через типажі (traits)

 // std::cmp::Ord / PartialOrd — стратегія порівняння
 let mut v = vec![3, 1, 4, 1, 5];
 v.sort();                              // стратегія за замовчуванням
 v.sort_by(|a, b| b.cmp(a));            // зворотній порядок
 v.sort_by_key(|x| std::cmp::Reverse(*x));

 // std::hash::Hasher — стратегія хешування
 use std::collections::HashMap;
 use std::hash::BuildHasherDefault;
 use rustc_hash::FxHasher;              // інша стратегія хешу
 let map: HashMap<_, _, BuildHasherDefault<FxHasher>> = HashMap::default();

 // std::io::Write — стратегія запису
 fn log(writer: &mut impl std::io::Write, msg: &str) {
     writeln!(writer, "{}", msg).unwrap();
 }
 log(&mut std::io::stdout(), "до файлу або stdout - однаково");
 log(&mut std::fs::File::create("log.txt").unwrap(), "те саме");

Висновки
1   Strategy — поведінковий GoF патерн, що інкапсулює алгоритм у окремий об'єкт


2   В Rust реалізується через trait: статично (generics) або динамічно (dyn Trait)


3   Статична диспетчеризація — нульові витрати, динамічна — гнучкість у runtime


4   Патерн присутній у стандартній бібліотеці Rust: sort_by, Write, Hasher та ін.


5   Застосовувати варто коли є кілька варіантів алгоритму, що можуть змінюватись

     Назва інструменту та версія: Claude (Anthropic), модель Claude Sonnet 4.6.
     Мета використання: консультація щодо вибору теми доповіді з-поміж
патернів GoF, що суміжні з мовою програмування Rust.
     Внесок     автора:    самостійне     формулювання        запитів,   оцінювання
запропонованих варіантів, прийняття рішення про вибір теми, перевірка та
коригування згенерованого коду, створення доповіді, створення презентації,
створення відеодоповіді.
     а)   Запит: «Які патерни GoF найкраще демонструються мовою Rust і мають
          практичне застосування у її стандартній бібліотеці або екосистемі? Мене
          цікавлять поведінкові патерни такі якStrategy, State та Iterator».
                Відповідь: Інструмент надав порівняльний огляд трьох патернів.
          Strategy було описано як такий, що природно відображається на
          типажі Rust: через узагальнені типи (статична диспетчеризація, zero-cost
          abstractions) або через «Box<dyn Trait>» (динамічна диспетчеризація).
          State було відзначено як цікавий через typestate pattern – кодування
          стану на рівні системи типів. Iterator описано як патерн, вбудований
          безпосередньо у мову через типаж «std::iter::Iterator».
     б)   Запит: «Чи доречно для доповіді про Strategy у Rust показати обидва
          варіанти диспетчеризації, маю на увазі статичну та динамічну?»
                Відповідь: Інструмент підтвердив доречність та запропонував
          структуру: спочатку показати визначення типажу і конкретних стратегій,
          потім окремо розглянути обидва варіанти контексту – «Sorter<S:
          SortStrategy>» та «DynSorter { strategy: Box<dyn SortStrategy> }» – з
          поясненням компромісів між ними.
